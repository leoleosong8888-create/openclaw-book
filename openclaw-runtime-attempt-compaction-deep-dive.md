# OpenClaw Runtime 해부: `attempt`, `runEmbeddedAttempt`, `compaction`이 왜 핵심인가

이 문서는 OpenClaw 런타임의 실제 실행 축을 **비개발자도 이해할 수 있게** 풀어쓴 설명입니다.

대상 질문:
1) 왜 `attempt` 개념이 필요한가?
2) 왜 `runEmbeddedAttempt`가 핵심인가?
3) 왜 `compaction`이 꼭 필요한가?

참고 파일(저장소 기준):
- `agents/pi-embedded-runner/run.ts`
- `agents/pi-embedded-runner/run/attempt.ts`
- `agents/pi-embedded-subscribe.ts`

---

## 0) 한 장 요약

```text
사용자 요청
  ↓
run.ts (상위 오케스트레이터)
  ↓
runEmbeddedAttempt(...)  ← 실제 1회 실행 단위(핵심)
  ↓
LLM + tools + session history 반영
  ↓
결과/오류/중단/타임아웃
  ↓
필요시 재시도(next attempt)
  ↓
최종 응답 + (필요시) compaction 반영
```

즉:
- `run.ts`는 **감독/조정자**
- `runEmbeddedAttempt`는 **현장 실행자**
- `compaction`은 **장기 메모리 품질 유지 장치**

---

## 1) 왜 `attempt`가 필요한가?

`attempt`는 “한 번의 실행 시도”를 의미합니다.

### 1-1. 현실 문제
실제 에이전트 실행은 항상 불확실성이 있어요.
- 도구 호출 실패
- 네트워크 지연
- 타임아웃
- 세션 파일 충돌/손상
- 중간 취소(사용자 인터럽트)

이걸 “무조건 한 번에 성공”으로 설계하면, 사용자 경험이 쉽게 깨집니다.

### 1-2. attempt의 역할
- 실패를 **구조화된 단위**로 관리
- 재시도 시 “어디서 실패했는지” 추적 가능
- 복구 전략(다시 시도/중단/상위에 에러 전달)을 명확히 분리

쉽게 말해:
> attempt는 런타임의 “복원력(resilience) 프레임”이다.

---

## 2) 왜 `runEmbeddedAttempt`가 핵심인가?

`runEmbeddedAttempt`는 실제로 대부분의 중요한 일을 수행합니다.

### 2-1. 이 함수가 담당하는 것
1. 실행 workspace/sandbox 확정
2. 스킬/부트스트랩 컨텍스트 로드
3. tools 집합 계산 및 정책 적용
4. 시스템 프롬프트 생성
5. 세션 락 획득 + 세션 매니저 준비
6. 히스토리/스트림 파이프라인 구성
7. Abort/timeout/subscription 연결
8. LLM turn 실행
9. compaction 대기/스냅샷 반영
10. 결과 조립 및 자원 정리

즉, 런타임의 “입출력 경계 + 상태 전이 + 안전장치”가 한 함수에서 만나요.

### 2-2. 왜 중요하냐
이 함수가 흔들리면:
- 응답 품질
- 도구 안전성
- 세션 일관성
- 장기 운영 안정성
이 동시에 무너집니다.

그래서 실무에서는 `runEmbeddedAttempt`를
> “OpenClaw 런타임의 심장”
으로 보는 게 맞습니다.

---

## 3) `run.ts`와의 관계 (상위/하위 역할 분리)

- `run.ts`: attempt를 호출하고, 재시도/종료 정책을 결정하는 상위 오케스트레이터
- `attempt.ts`: 실제 실행 1회를 끝까지 밀어붙이는 실행기

이 분리가 중요한 이유:
- 정책과 실행을 분리해 유지보수성 확보
- 재시도 전략 변경 시 상위 레이어만 수정 가능
- 실행기 복잡도를 상위 정책으로부터 보호

---

## 4) `compaction`이 왜 필요한가?

LLM 컨텍스트는 유한합니다. 대화가 길어질수록 세션 히스토리가 커져요.

### 4-1. compaction이 없으면 생기는 문제
- 토큰 폭증 → 비용 증가
- 컨텍스트 초과 위험
- 중요한 정보가 뒤로 밀려 정확도 하락
- 반복/장황 응답 증가

### 4-2. compaction의 본질
`compaction`은 과거 대화를 그냥 버리는 게 아니라,
- 핵심 의사결정
- 사용자 선호
- 진행 중 작업 상태
를 요약/정리해서 다음 턴에 유지합니다.

즉,
> “기억을 삭제”가 아니라 “기억을 압축해 품질을 유지”하는 과정.

### 4-3. 운영 관점 가치
- 긴 세션에서도 일관성 유지
- 비용/지연 최적화
- 에이전트가 ‘이전 맥락’을 더 안정적으로 활용

---

## 5) subscribe 계층이 하는 일 (왜 따로 있나)

`pi-embedded-subscribe.ts` 계층은 스트리밍/이벤트 전달과 연결됩니다.

역할:
- 런타임 이벤트를 외부(채널/UI/세션)로 전달
- 진행 상태, 중간 출력, 완료/실패를 일관된 형식으로 브로드캐스트

분리 이점:
- 실행 로직(`attempt`)과 전달 로직(`subscribe`) decoupling
- 채널이 달라도 실행기 코어는 동일하게 유지

---

## 6) 실무 디버깅 포인트

### 6-1. 시도 단위로 로그 보라
- “attempt 시작 → 실패 지점 → 정리” 순으로 보면 원인 파악이 빨라짐

### 6-2. 세션 락/파일 무결성 먼저 확인
- 이상 현상의 상당수가 세션 상태 충돌에서 시작

### 6-3. compaction 시점 전후 품질 비교
- 답변이 갑자기 맥락을 잃으면 compaction 결과물을 의심

### 6-4. timeout/abort 경계 확인
- 사용자 체감상 “멈춤” 이슈는 대부분 여기서 발생

---

## 7) 최종 결론

- `attempt`는 실패를 다루기 위한 **복원력 단위**
- `runEmbeddedAttempt`는 런타임의 실제 실행을 책임지는 **핵심 엔진**
- `compaction`은 긴 세션에서 정확도·비용·일관성을 지키는 **필수 메커니즘**

세 가지를 함께 보면 OpenClaw 런타임은
> “한 번 잘 답하는 시스템”이 아니라,
> “오래 안정적으로 운영되는 시스템”으로 설계돼 있다는 걸 알 수 있습니다.
